## 1. 목표

자연어로 물리 문제를 설명하면 → 적절한 COMSOL 모델을 제안하고 → 필수 입력값을 폼으로 받아 → COMSOL을 실행하고(내 PC 솔브 **또는** .mph 반출 → 오프라인 서버 솔브 → 업로드) → 결과를 자연어 대화로 탐색할 수 있는 PWA.

**사용 시나리오 예시**

> 사용자: "실리콘 태양전지인데 반사방지막 두께를 바꿔가며 효율 변화를 보고 싶어"
> 앱: "Si 태양전지 템플릿(Semiconductor + Wave Optics 결합)을 제안합니다. 필수 입력 8개가 필요합니다" → 폼 표시
> 사용자: 입력값 채움 → 실행 → 진행률 표시
> 앱: "ARC 80nm에서 효율 최대 21.3%…" + J-V 곡선 플롯
> 사용자: "파장별 흡수율도 보여줘" → 앱이 기존 해에서 추가 결과 추출 + 해석

## 3. 핵심 설계 결정

### 결정 1 — COMSOL 연동: MPh (Python)

| 방식 | 평가 |
|---|---|
| **MPh (Python, 권장)** | 오픈소스 COMSOL Java API 래퍼. .mph 로드→파라미터 변경→솔브→결과 평가/내보내기 커버. FastAPI와 같은 언어/프로세스. v1.3.1, 2026-05에도 업데이트 확인됨 |
| Java API 직접 | 가장 강력하나 별도 컴파일·프로세스 관리 필요. MPh 실패 시 대안 |
| 명령줄 배치 (`comsolbatch`) | 파라미터 스윕만 가능, 결과 추출·대화형 후처리 불가 |
| LiveLink for MATLAB | 별도 유료 라이선스 필요 → 배제 |
| COMSOL Server | 앱 배포용 별도 제품, 미보유 → 배제 |

⚠️ **검증 필요**: 사용자 COMSOL 버전(6.3? 6.4?)과 MPh 호환성 — Phase 0에서 실측이 첫 작업.

### 결정 2 — 모델 생성: 템플릿 우선 (가장 중요한 결정)

LLM이 COMSOL API 코드를 자유 생성하는 방식은 **실패율이 높고 오류 검증이 불가능**하므로 배제합니다. 대신:

1. 검증된 **파라미터화 .mph 템플릿 라이브러리**를 사람이(우리가) 만들어 두고
2. LLM의 역할은 ① 자연어 → 템플릿 선택 ② 필수 입력 스키마 제시 ③ 사용자 입력 → 파라미터 매핑으로 제한
3. 템플릿 밖 요청은 "현재 미지원" + 어떤 템플릿이 가까운지 안내 + 요청 로그 축적 → 템플릿 추가 근거

신뢰성은 템플릿이 보장하고, 편의성은 LLM이 제공하는 구조입니다. 커버리지는 템플릿을 늘려 확장합니다.

### 결정 3 — AI 서비스: Claude 단일 기본 + "모델 제안" 단계만 비교 모드(옵션)

요청하신 판단입니다. **전 단계 멀티모델 상시 비교는 비권장** — 비용·지연이 2~3배가 되고 유지보수가 복잡해지는데, 파싱·폼 생성 같은 단계는 모델 간 차이가 작아 비교 가치가 낮습니다. 대신:

- **어댑터 계층**을 처음부터 두어 provider 교체·추가가 설정 한 줄이 되게 설계 (lock-in 없음)
- **시작은 Anthropic 키 1개**. 단계별 모델 티어:

| 파이프라인 단계 | 모델 | 이유 |
|---|---|---|
| 자연어 파싱, 폼 스키마 | Claude Sonnet 5 | 구조화 출력 충분, 저비용 ($2/$10 /1M tokens, 2026-08까지 프로모션가) |
| 모델 제안, 결과 해석 | Claude Opus 4.8 ($5/$25) | 판단·물리 추론이 필요한 단계. Sonnet 5로 시작해 품질 미달 시 상향도 가능 |

- **비교 모드 (Phase 4, 옵션)**: 판단이 갈리는 "모델 제안" 단계에만 GPT-5.5 ($5/$30) / Gemini 3.x Pro ($2/$12) 병렬 호출 → 제안 카드를 나란히 표시 → 사용자가 선택. 동일 JSON 스키마로 출력을 강제하므로 구현 비용이 낮음
- 공정성 고지: 저는 Claude라 이해관계가 있습니다. 다만 어댑터 구조라 기본 provider는 언제든 교체 가능하고, 비교 모드가 검증 수단이 됩니다
- ⚠️ 가격은 2026-07 웹 검색 기준(2차 출처 포함) — 키 발급 전 공식 페이지에서 확인 필요

## 4. 아키텍처

```
[폰/PC 브라우저 · PWA]
   Next.js (App Router, TS)
   채팅 UI · 동적 폼 · 결과 뷰어(플롯/표) · 작업 상태 · 이력
        │ HTTPS (로컬: localhost / 원격: Tailscale)
        ▼
[사용자 PC · FastAPI 백엔드]
   ├─ LLM 오케스트레이터 ── LLM 어댑터 ──→ Anthropic API (+ OpenAI/Google, Phase 4)
   ├─ 스펙 관리 (ModelSpec JSON + Snapshot 이력, SQLite)
   ├─ 작업 큐 (동시 솔브 1개, SSE로 진행 상황 푸시)
   └─ COMSOL 워커 ── MPh ──→ COMSOL Multiphysics (동일 PC)
                              └─ 템플릿 .mph 라이브러리 / 결과물 (PNG·CSV·mph)
        │ ↓ 미솔브 .mph + 실행 스크립트 다운로드 / ↑ 솔브된 .mph 업로드
        ▼
[오프라인 서버] ←USB→ comsolbatch로 스터디 실행 (앱·인터넷 불필요)
```

단일 사용자이므로 큐·DB·인증 모두 최소 구성(SQLite, 토큰 1개짜리 간단 인증)으로 시작합니다.

## 5. 파이프라인 상세 (5단계)

| 단계 | 내용 | 신뢰성 장치 |
|---|---|---|
| S1 해석 | 자연어 → 구조화 추출 (물리, 지오메트리, 재료, 경계조건, 원하는 출력) | 신뢰도 색상, warnings 표시 (기존 정책) |
| S2 제안 | 템플릿 매칭 + 선택 근거 + 대안 제시 | 템플릿 enum 안에서만 선택 |
| S3 입력 | 템플릿의 필수 입력 JSON Schema → 폼 자동 생성 | 폼-only 편집, 단위 변환(SI 정규화), **누락은 누락으로 표시 — LLM이 임의값 채우기 금지(시스템 프롬프트 + 스키마 검증 이중 장치)** |
| S4 실행 | 스펙 검증 → 작업 생성 → **실행 대상 선택**: ① 내 PC에서 솔브(MPh, 진행률/취소) ② 미솔브 .mph 다운로드(오프라인 서버용) | 실행 전 파라미터 요약 확인 화면 |
| S5 결과 대화 | 백엔드가 추출한 수치 요약 + 플롯 PNG를 LLM에 전달(멀티모달) → 자연어 해석. 후속 요청("파장별 흡수율 보여줘")은 **제한된 결과 연산 DSL**(evaluate / plot / sweep-table / export)로 변환 후 실행 | LLM이 자유 코드를 실행하지 않음. DSL 연산만 허용 |

### 오프라인 서버 실행 경로 (S4의 ②)

1. 앱이 템플릿 + 입력 파라미터로 모델을 구성하고 스터디까지 설정한 **미솔브 .mph를 저장** → 다운로드 제공. 서버용 실행 스크립트 동봉 (`run.bat`/`run.sh`: `comsolbatch -inputfile model.mph -outputfile model_solved.mph`)
2. USB 등으로 서버에 복사 → 스크립트 실행. 서버에는 앱·인터넷 불필요
3. 솔브된 .mph를 내 PC로 회수 → 앱에 **업로드** (대용량은 지정 폴더 가져오기 방식 병행) → 백엔드가 MPh로 열어 결과 추출 → S5 결과 대화는 로컬 솔브와 동일

⚠️ 전제: 서버 COMSOL 버전이 내 PC와 같거나 높아야 하며(구버전은 신버전 .mph를 못 엶), Wave Optics·Semiconductor 모듈 라이선스가 서버에도 필요.

## 6. 데이터 모델 (요약)

- **ModelSpec**: schema_version(semver), 템플릿 ID, 파라미터(값+입력 단위+SI 정규화값), 상태(누락/가정/확정), warnings
- **Snapshot**: 스펙 변경 이력 (기존 정책)
- **Job**: 스펙 snapshot 참조, 실행 모드(local | external), 상태(대기/실행/**반출됨/업로드 대기**/완료/실패/취소), 로그
- **ResultArtifact**: 유형(스칼라/표/플롯), 파일 경로, 생성 DSL 명령

| ID | 템플릿 | 물리 | 필수 입력 예시 | 주요 출력 |

제가 파라미터화 사양을 설계하고, COMSOL 안에서의 템플릿 조정은 사용자와 협업합니다(Phase 1).

## 8. 개발 로드맵 (작은 단위, 단계별 완료 기준)

| Phase | 내용 | 완료 기준 |
|---|---|---|
| **0. 리스크 스파이크** | 사용자 PC에서 MPh 설치 → 기존 .mph 열기 → 파라미터 변경 → **미솔브 저장(save)** → 솔브 → 스칼라 평가 + 플롯 PNG 내보내기 → **솔브된 파일을 다시 열어 재솔브 없이 결과 추출** | 7개 동작 전부 성공 로그. **실패 시 아키텍처 재검토(Java 배치 대안)** |
| 1. 실행 골격 | FastAPI + SQLite + 작업 큐 + 템플릿 1개(T1 또는 T2), 실행 두 경로(로컬 솔브 / .mph 다운로드→업로드) (LLM 없음) | curl로 ① 로컬 솔브 결과 수신 ② 다운로드→수동 솔브→업로드→결과 추출 성공 |
| 2. LLM 파이프라인 | S1–S3: 자연어 → 제안 → 폼 (Next.js UI 포함) | 예시 문장 10개로 테스트, 임의값 채움 0건 |
| 3. 결과 대화 | S5: 요약+해석, 결과 DSL 후속 질의 | 시나리오(1절) 전체가 실제로 동작 |
| 4. 비교 모드 | 제안 단계 멀티모델 병렬 + 선택 UI (옵션) | 2개 이상 provider 나란히 표시·선택 |
| 5. PWA 마감 | manifest, service worker, 설치, Tailscale HTTPS, 이력 UI | 폰에 설치되어 외부에서 작업 제출→결과 확인 |

각 Phase 종료 시 변경 내용·남은 문제를 정리해 보고합니다(기존 작업 원칙).

## 9. PWA 배포 방안

- PWA 설치 조건은 HTTPS(또는 localhost) → **Tailscale**로 사용자 PC에 어디서나 HTTPS 접속 (무료, 포트 개방 불필요, 1인 사용에 적합)
- Next.js manifest + service worker(Serwist 계열 — ⚠️ 구현 시점에 최신 라이브러리 재확인)
- 오프라인 범위: 앱 셸 + 저장된 이력 열람까지. 솔브는 백엔드 필요
- 외부 배포(Vercel 등)는 불필요 — 백엔드가 COMSOL PC에 있어야 하므로 전부 로컬 서비스

## 11. 리스크와 완화

- **템플릿 커버리지 기대 차이**: 초보 사용자는 "아무 모델이나" 기대 → 첫 화면에 지원 범위 명시, 미지원 요청은 정중히 안내 + 로그
- **LLM 환각**: 폼-only 편집 + 임의값 금지 + 스키마 검증 + 실행 전 확인 화면의 4중 장치
- **솔브 시간/메모리**: 실행 전 예상 규모 경고, 취소 기능, 동시 1작업 제한
- **단일 PC 의존**: PC가 꺼지면 서비스 중단 — 개인용이므로 수용

## 14. v0.3 확장 아키텍처 — "정해진 케이스"에서 "똑똑한 반응"으로 (2026-07-06)

동기: 케이스 추가 때마다 수동 자료 조사·코드 수정이 필요한 구조는 확장이 불편함.
요구: 자연어 우선 흐름 / 측정 데이터 반입(AI 포맷 변환) / 자료 최신화·버전 비교 / 참고문헌 추적 / 케이스 파생·수정.

### 핵심 전환: 케이스 = 코드 → 케이스 = 데이터

| 구성요소 | 내용 |
|---|---|
| **케이스 정의(JSON)** | 레이어 스택(재료 참조+두께)·도핑·광생성 대상·스윕·출력을 선언. 파이썬은 이를 해석하는 **제네릭 1D 스택 빌더** 하나만 유지 |
| **재료 라이브러리(materials.json)** | 재료별 물성 세트. 각 값에 출처(ref id)와 **버전**. 문헌값과 사용자 측정값이 공존 |
| **참고문헌 저장소(references.json)** | ref id → 서지정보·URL·사용처 역링크. UI ⑤ 참고문헌 탭에서 열람 |
| **데이터셋 레지스트리(기존)** | 곡선형 데이터(n,k·스펙트럼)도 출처·버전 체계로 통합 |

가능해지는 것: 케이스 파생 = JSON 복제·수정("C60을 PCBM으로 바꿔줘"를 자연어로) / 물성 버전 A·B로
같은 케이스를 돌려 결과 비교 / 측정 데이터를 라이브러리에 등록해 즉시 사용.

### 자동화의 정직한 경계

- 문헌 검색·최신화·측정데이터 변환은 **복붙 모드**(구독 챗) 또는 API 모드로: 앱이 목적별 프롬프트를
  생성하고 AI 응답(JSON)을 검증해 등록. 앱이 스스로 웹을 크롤링하지 않음
- LLM이 만든 케이스·물성은 반드시 스키마 검증 + 사용자 검토 후 실행 (임의값 금지 유지)
- 유연성 범위는 "1D 다층 반도체 + 광생성" 물리 안 — 2D/3D·타 물리는 별도 빌더가 필요한 확장

| M | 내용 | 상태 |
|---|---|---|
| M1 | 제네릭 1D 스택 빌더 + materials/references 저장소, C60/BCP 케이스(v1 사양)를 첫 데이터 케이스로 | 진행 중 |
| M2 | ⑤ 참고문헌 탭 + 물성 버전 관리 + 버전 선택 실행·비교 | |

| M4 | 측정 데이터 인테이크: 업로드 → AI 검사·표준 포맷 변환 → 라이브러리 등록 | |
| M5 | 자료 최신화: 업데이트 확인 프롬프트 → 새 버전 등록 → 신구 비교 실행 | |
