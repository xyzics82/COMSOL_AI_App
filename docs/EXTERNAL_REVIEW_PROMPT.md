=== 프롬프트 시작 ===

너는 시니어 리뷰어다. 반도체 소자 시뮬레이션(COMSOL Semiconductor + Wave Optics)과
Python/FastAPI 웹앱 양쪽에 익숙하다고 가정한다. 아래 프로젝트를 종합 검토하고
구체적인 수정안을 제시하라. 답변은 전부 한국어.

## 1. 프로젝트 개요

- 목적: 자연어로 물리 문제를 받아 → 케이스(폼) → COMSOL 모델 생성 → 솔브 → 지표/그래프까지
  자동화하는 1인용 연구 웹앱. 궁극적으로 연구 AI 에이전트의 구성요소.
- 대상 물리: 페로브스카이트(MAPbI3) IBC(후면 깍지 전극) 태양전지. 2D 검증 후 3D 단위셀
  (핑거 끝단 포함)로 확장. 광생성은 평판 파동광학 G(z) 주입(TMM 대조 검증됨).
- 스택: FastAPI(127.0.0.1) + 단일 워커 큐 + SQLite 작업기록 / MPh 1.3.1 → COMSOL 6.4
  (Windows, 라이선스 USB 동글) / 프론트는 바닐라 JS 단일 index.html.
- 멀티엔진: COMSOL 외 SCAPS-1D·IonMonger(MATLAB)·Driftfusion(MATLAB)·Solcore(Python)·
  Quantum ESPRESSO(WSL)를 같은 케이스/작업 프레임에 디스패치.
- 운영 형태: 개발 PC(동글 없음, 현재)와 계산 서버(동글 있음)를 git으로 오간다.
  긴 배치는 서버에서 밤샘 실행 → 로그를 사람이 복사해 개발 세션에 회신.
- 핵심 제약: **API 키 불필요가 기본** — 자연어 파싱·케이스 생성·모델 수정은 "복붙 모드"
  (자가완결 프롬프트를 만들어 사용자가 외부 챗 AI에 붙여넣고 JSON 답변을 앱에 다시
  붙여넣으면 검증 후 반영)가 1급 경로다.

## 2. 실측으로 확정된 사실 (함부로 "수정"하지 말 것)

아래는 실제 배치 실행(2026-07-08~13, 100조합 전멸 → 30조합 80% → 15조합 14/15 회수)으로
확정한 경험적 사실이다. 이와 충돌하는 수정을 제안하려면, 왜 기존 판단이 틀렸는지
근거(수치·문헌·솔버 이론)를 먼저 제시하라. 추측으로 뒤집지 마라.

1. 3D 실패의 주 원인은 메시-뉴턴 상성: 같은 물리에서 hmax만 120→150/100/135/90nm로
   바꾸면 발산 셀이 수렴한다(누적 11건 이상 구제). 코드의 "메시 사다리"가 그 구현.
2. hmax 80nm는 끝단+IDL 3D에서 평형(Study 1)이 오히려 발산한다. 기본 120nm.
3. 스윕 0→2V는 강순방향에서 다발 발산. 기본 상한 1.3V + 0.9V 이후 스텝 절반.
4. 발산 전압은 1.05V, 1.13V 부근에 집중(특히 W=2um 계열은 3/3회 1.13V 재현).
   Voc≈1.05V 이후이므로 부분 데이터로 지표는 유효. 구조적 원인은 미해명(검토 대상).
5. 핑거 끝단(팁<셀길이)×IDL 2nm의 z-계단은 Swept 메시를 파괴한다. 해법: IDL 밴드를
   z-전구간 2조각으로(단위셀), 배열은 흡수층 블록을 z_tip에서 분할.
6. IDL(계면 결함층) 2nm 등가층: S[cm/s] → τ_IDL = d/S, 벌크 SRH와 가산. 2D에서 검증.
7. 발산 직후 COMSOL 모델은 평가(evaluate)를 거부하기도 한다 — 저장 후 재로드하면 평가
   가능. 부분회수 로직이 이 순서로 짜여 있다.
8. uvicorn --reload는 솔브 중 백엔드 파일 편집 시 진행 중 작업을 죽인다(3회 겪음).
9. IonMonger parameters.m은 `function params = parameters(iter)` 형태라 원본 템플릿
   정규식 치환으로 생성(29필드). Driftfusion dfana.calcJ는 "sub"만 유효("whole"은
   소스 버그). Solcore TMM은 파장 벡터화 필수. 모두 실코드 대조·실행으로 확정.
10. .bat과 SCAPS 스크립트는 ASCII 전용(한글 주석 → cp949/인코딩 사고 2회).

## 3. 작업 원칙 (수정안도 반드시 따를 것)

- 임의값 금지: 출처 없는 물성·상수를 지어내지 마라. 필요하면 "누락 — 사용자 확인 필요"로 표시.
- 불확실한 COMSOL/MPh API는 사용하되 "검증 필요" 딱지를 명시하라(이 저장소는 실행
  검증 전 코드를 그대로 신뢰하지 않는다).
- 복붙 모드(무-API) 경로를 깨는 변경 금지. API(anthropic) 의존은 항상 선택 사항.
- 작은 단위 패치 우선. 대규모 리팩터링을 제안하려면 단계별 마이그레이션 계획을 붙여라.
- 보안: .env(API 키)를 코드/로그/커밋에 노출하는 패턴이 보이면 지적하라.

## 5. 검토 요청 항목 (우선순위순)

A. **수렴·솔버 전략**: 메시 사다리(120→150→100→135→90 + 최후 스윕 미세화)의 논리,
   Voc 기준 부분수락(last_V ≥ min(1.1, v_max)), best-partial 보존/재로드 회수의 허점.
   1.13V 재현 발산의 원인 가설(고주입에서 IDL 등가층의 캐리어 밀도 폭주? 접점 경계?
   램핑/continuation 대안? COMSOL Semiconductor의 세그리게이티드 vs 풀커플드 선택?)과
   실험 가능한 검증 방법을 제시하라. 사다리보다 원리적인 해법(초기해 연속법, 전압
   스텝 적응 제어, 감쇠 뉴턴 파라미터)이 MPh로 구현 가능하면 코드로.
B. **물리 타당성**: FF 0.43~0.55로 낮은 원인(직렬저항 없는 모델에서 이 값이 타당한가,
   수송층 저항? 게이트 없는 IBC 기하의 본질?), 수집효율 100.7%/101.6% 같은 >100% 값
   (jsc_ref 정의 문제인지), IDL 등가층 근사의 한계, G(z) 1D 주입이 3D 기하에서 갖는 오차.
C. **코드 품질**: comsol_cases.py의 _run_ibc3d(긴 함수, 상태 변수 다수), timeline.py 파서
   견고성, llm_propose.py 검증 로직, main.py API 설계, 예외 처리 일관성.
D. **프론트/UX**: index.html 단일 파일 구조의 유지보수성, 검토→솔브→수정 워크플로의
   빈틈(경합 조건, 탭 전환 시 폼 리셋 등), 타임라인 SVG 렌더링.
E. **아키텍처**: 단일 워커 큐의 한계, 서버/로컬 이원 운영(git 수동 동기화)의 개선안,
   멀티엔진 추상화의 적절성.
